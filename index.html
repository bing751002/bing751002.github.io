本文撰寫時的OTel版本為: 1.25

OpenTelemetry on .net

在.net framework中，已提供logging，metrics以及activity APIs 實作OTel的標準，代表OTel不用再額外提供APIs，只要使用原生的APIs即可。.Net OTel以下列的方式實作了OTel的標準

  • Microsoft.Extensions.Logging.ILogger (Logging)
  • System.Diagnostics.Metrics.Meter (Metrics)
  • System.Diagnostics.ActivitySource and System.Diagnostics.Activity (Tracing)

OpenTelemetry in .NET is implemented as a series of NuGet packages that form a couple of categories:

  • Core API
  • Instrumentation - these packages collect instrumentation from the runtime and common libraries.
  • Exporters - these interface with APM systems such as Prometheus, Jaeger, and OTLP.

第一種方式為在使用Automatic的情況下塞入自定義的Instrumentation，這種方式可以在僅修改少量原始碼的情況下，將自定義的Instrumentation塞入到Auto Instrumentation中。

Nest Activities

指的是多個Activities並且有父子(Parent-Child)關係，這種情況下，會有一個root activity，並且會有一個parent activity，parent activity會有一個或多個child activity，child activity也可以有child activity，如此一來就會形成一個activity tree。

unit-test-flow

Posted on 2023-07-29 Edited on 2023-10-30

1.開發者寫完程式碼與UT。
2.在自己電腦上跑UT。
在自己電腦上跑UT，是部門不認的UT。
沒人知道你自己電腦的環境與有沒有動手腳。

  1. Commit and push 到 repository 開發分支。

  2. 啟動 CI ，CI有個stage會去跑開發者的UT。

    由於UT已經不在開發者的電腦或環境跑了。

所以有許多優點：

a. 環境是獨立的，而且通常設計成接近 release後的環境。比較容易提早發現問題。

b. 開發者有沒有做好UT，Pass UT，是有自動記錄，而且自己沒有權限修改的。避免了前

  1. 所有CI流程都過了，UT過了，開發者以外的工程師或主管，才開始審核程式碼 code r
    eview。(正常情況，至少兩人)

這時審核的人，系統都會自動紀錄。
比較大的公司也會有規定，或慣例該review哪些重點。
6. Code review 過了，系統才會自動 merge到 “開發”分支。(因為還沒給QA測過，沒辦
法release)

  1. QA 測試前，先再次跑CI流程，包含UT，確認開發部門有按照基本品質要求走。(避免
    被Dev部門黑)。拉取程式與自己的測試程式，在接近生產環境的設備上測試。

  2. QA測試出報告，有問題，提issue修改。沒問題，上系統或出Mail說驗證通過。(為品
    質背書)

  3. 程式碼品質Ok了，要將 dev merge到release分支。開發者根本沒這權限。只有技術的
    owner或 Tech lead 有merge權限。有merge權限的人，要對這程式碼品質負責。

以上的流程，已經簡化蠻多細節了，而且變化很多，同家公司不同部門細節也不同，但大
原則沒變。

看似複雜冗長，其實大多機器自動化去做，大多寫程式就能完成自動化，習慣了就好。兩
個星期跑release 一個線上版本很正常。

線上系統出問題，誰有責任：
開發者，owner，開發者主管，測試QA工程師，QA工程師主管，PM都可能會有責任。
大家不是靠嘴去爭的，拿出Log與證據來討論吧。

Introduction

kubernetes 俗稱 k8s, 是一個用來管理容器 (container) 的系統，想像一下今天在管理docker時, 總共有100個container 要管理，包含部署、scaling、availability、監控、log管理、網路管理以及設定等，以上繁瑣的事項如果都要靠人為的方式管理那十之八九會有問題。為了讓以上的流程更為順利，K8S用了一個更高level的方式去管理所有的container 像是 以YAML方式設定要執行的container的內容, 對要執行的jobs做scheduling等等。

k8s 基本元件

  • Pod
    k8s的最小元件 主要用來裝載service (container)的元件. 一個pod裏面可以裝載多個container。但若是多個container也意味著會有多個應用(application) 在同個Pod裏面，因此通常理想上一個Pod只裝載一個container為主
    每個Pod會有描述這個Pod的yaml檔案
  • Node
    k8s的最小實體元件，可以是一台VM或是一台實體主機(ex. 筆電)，每個Node內主要有三個元件. kubelet, kube-proxy, container runtime
  • Kubelet
    該Node的管理程式，負責管理此node的狀態以及跟外界(master node)的溝通

Master Node
負責管理一般node的node. 算是k8s的管理中心，用來管理各個node。通常master node會分散在不同的機器上，以免Node損壞導致無法運行。主要會有以下幾個元件

  • API server
  • Scheduler
  • Controller-manager
  • etcd
